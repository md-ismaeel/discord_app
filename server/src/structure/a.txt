Key Flows & Connections
1. User & Identity (User)
The User model is central. It stores authentication details and profile data.

Relationships:
Users can be friends (friends array of User IDs).
Users can block others (blockedUsers).
2. Servers & Communities (Server, ServerMember)
A Server is a community. Users don't belong directly to a server; they are linked via a ServerMember document.

Why this design? This allows storing server-specific user data (nickname, roles, mute status) separate from the global user profile.
Flow: User -> ServerMember -> Server.
3. Channels & Messaging (Channel, Message)
Messages belong to a Channel, and a Channel belongs to a Server.
Flow to find messages: Server -> Channels -> Messages.
Messages reference the User (author) and the Channel.
4. Roles & Permissions (Role)
Roles are defined per Server.
A ServerMember has an array of Role IDs.
Logic: To check permissions, you fetch the ServerMember for a user in a specific server, populate their roles, and check the permissions flags on those roles.
5. Private Messaging (DirectMessage)
Independent of servers.
Links two User documents directly (sender and receiver).
6. Invites (Invite)
Generated for a specific Server by a User.
Used to create a new ServerMember entry for the joining user.
